## 2. Token 节能技术深度剖析

### 2.1 技术全景图

1234567KF/ccmvp 的 Token 节能体系由以下 **7 大机制** 构成：

```
┌────────────────────────────────────────────────────────────────┐
│                     Token 节能七层塔                              │
├────────────────────────────────────────────────────────────────┤
│  L7: 模型智能路由    │ kf-model-router — Pro↔Flash 分阶段切换      │
│  L6: A2A 通信压缩   │ λ-lang — Agent 间 3x 压缩消息              │
│  L5: 阶段智能跳过    │ CCP Skip — 变更小时跳过不必要阶段            │
│  L4: 技能按需加载    │ skill-loader — 非活跃技能→25 token stub     │
│  L3: 长上下文预热    │ 预热 PRD/Spec 触发 KV Cache checkpoint      │
│  L2: 共享前缀缓存    │ 所有 Agent 前200-500 token 逐字相同          │
│  L1: 输出内容压缩    │ tokenforge + lean-ctx — 80-99% 压缩        │
└────────────────────────────────────────────────────────────────┘
```

### 2.2 L1：输出内容压缩（tokenforge + lean-ctx）

#### tokenforge.cjs — LLM Token 压缩引擎

纯 Node.js 实现，无需 Rust 依赖，提供 **4 种压缩类型**：

| 压缩类型 | 目标内容 | 核心策略 |
|----------|----------|----------|
| **output** | Shell/Terminal 输出 | ANSI 剥离 → 空行折叠 → 重复行去重 → 栈跟踪折叠 → 长行截断 → 行数限制 |
| **json** | API 响应 / JSON | 深度限制（3/5/8）→ 数组采样 → 大对象摘要 → 键压缩 |
| **code** | 源代码文件 | 导入/签名保留 → 函数体折叠 → 块注释移除 → AST 感知 |
| **auto** | 自动检测 | 基于内容特征自动选择最适压缩器 |

**三种压缩级别**：

| 级别 | 最大行数 | 适用场景 |
|------|----------|----------|
| `light` | 300 行 | 需要较高保真度 |
| `medium` | 120 行 | 默认推荐 |
| `aggressive` | 60 行 | 极致压缩 |

**实际效果**（来自海报数据）：

| 操作 | 原始输出 | tokenforge 输出 | 节省 |
|------|---------|-----------------|------|
| `vitest run` | 102,199 chars | 377 chars | **99.6%** |
| `npm test` | 25,000 chars | 2,500 chars | **90%** |
| `git status` | 7,500 chars | 1,500 chars | **80%** |
| `ls / tree` | 2,000 chars | 400 chars | **80%** |

**关键设计亮点**：
- 通过 `tokenforge-hook.cjs` 实现 **PreToolUse Hook** 自动注入，用户无需手动调用
- 智能跳过交互命令（git push/commit、npm install、编辑器等），避免误压缩
- 内置统计系统（`tokenforge stats`），追踪累计节省量

#### lean-ctx — 上下文工程层

第三方 MCP 工具（作者 garrytan，v3.4.7），6873+ 安装量。提供 71 个 MCP 工具、10 种读取模式、95+ 压缩模式。

**核心工具替代策略**：

| 优先使用 | 替代原生 | 原因 |
|----------|----------|------|
| `ctx_read(path, mode)` | Read / cat | 缓存 + 10 种读取模式，重读仅 ~13 tokens |
| `ctx_shell(command)` | Shell / bash | git/npm/cargo 输出模式压缩 |
| `ctx_search(pattern, path)` | Grep / rg | 紧凑、Token 高效的结果 |
| `ctx_tree(path, depth)` | ls / find | 紧凑目录映射 |
| `ctx_edit(...)` | Edit | 无需 Read 的搜索替换 |

**10 种 ctx_read 模式**：`auto`, `full`, `map`, `signatures`, `diff`, `aggressive`, `entropy`, `task`, `reference`, `lines:N-M`

### 2.3 L2：共享前缀缓存（KV Cache 优化）

这是整个节能体系中**成本杠杆最高**的机制。基于 DeepSeek API 的 KV Cache 特性：

```
缓存命中：  ¥0.025 / M tokens（Pro）   ¥0.02 / M tokens（Flash）
缓存未命中：¥3.000 / M tokens（Pro）   ¥1.00 / M tokens（Flash）
─────────────────────────────────────────────────────────
价差倍率：  120x                         50x
```

**三级缓存架构**：

```
L1: 共享前缀缓存（200-500 tokens）
  └─ 所有技能/agent 的 system prompt 前缀逐字相同
  └─ 命中率 > 90%，成本降低 67%

L2: 长上下文预热（长文档一次性加载）
  └─ PRD/Spec/进度文件在首请求中预热
  └─ 后续请求命中 KV Cache checkpoint

L3: 技能内容按需加载（skill-loader 控制）
  └─ 当前阶段 required 技能 → 完整加载
  └─ 当前阶段 recommended/contextual → 按需触发
  └─ 非当前阶段技能 → 元数据 stub（~25 tokens/技能）
```

**缓存优化策略核心要点**：

1. **全局 System Prompt 统一**：所有技能的 system prompt 前 200-500 token **逐字相同**（包括空格、换行、标点）
2. **串行模式优势**：通用 IDE 串行执行时，首请求缓存落盘后，后续请求天然命中（优于真并行模式）
3. **TTL 管理**：DeepSeek KV Cache TTL 为 5 分钟，须在间隔内发送后续请求
4. **监控指标**：通过 `usage.prompt_cache_hit_tokens` 和 `usage.prompt_cache_miss_tokens` 实时监控命中率

**L3 按需加载压缩效果**（24 技能 / ~1,300KB / ~370K tokens 为例）：

| 加载模式 | 加载技能数 | 上下文 tokens | 压缩率 |
|----------|-----------|-------------|--------|
| 全量加载 | 24 | ~370K | 0% |
| 按需加载 (stage-5) | 5-8 | ~45K + 400（元数据）| **~88%** |
| 按需加载 (stage-0) | 2-3 | ~15K + 525（元数据）| **~96%** |

### 2.4 L5：智能阶段跳过（CCP Skip）

当变更文件 < 3 个时，自动跳过不必要的子阶段（编码/测试等），每跳过一个阶段节省 10K-15K tokens。

### 2.5 L6：A2A 通信压缩（λ-lang）

Agent-to-Agent 原生通信语言，340+ 原子指令，7 个域（a2a/evo/code/...），将原生 JSON/自然语言的 Agent 间消息压缩至 1/3。

### 2.6 L7：模型智能路由（kf-model-router）

虽然属于"成本优化"而非纯"Token 优化"，但同样重要：

- 计划/架构任务 → DeepSeek Pro（¥3/M input）
- 执行/编码/测试任务 → DeepSeek Flash（¥1/M input）
- 后端支持 Kimi for Coding、MiniMax M2.5 等多模型
- 包含断路器、令牌桶限流、降级链等安全机制

### 2.7 综合收益估算

| 场景 | 无优化 | 有优化 | 节省 |
|------|--------|--------|------|
| 总输入 tokens（3 Agent 串行）| 3 × 8K = 24K | 1 × 8K + 2 × 0.5K(缓存) + 2 × 2K(suffix) = 13K | **46%** |
| 成本 (Pro) | 24K × ¥3/M = ¥0.072 | 8K × ¥3 + 5K × ¥0.025 = ¥0.0241 | **67%** |
| 成本 (Flash) | 24K × ¥1/M = ¥0.024 | 8K × ¥1 + 5K × ¥0.02 = ¥0.0081 | **66%** |

**海报声称总体效果**：80% Token 节省率，成本降至原来的 1/5。

## 3. 国际高星开源项目调研

### 3.1 Headroom ⭐ 8,500+

**仓库**：[chopratejas/headroom](https://github.com/chopratejas/headroom)

**定位**：上下文压缩代理层，位于 AI Agent 和 LLM Provider 之间

```
Your Agent/App → Headroom (本地) → LLM Provider
                   ├─ CacheAligner
                   ├─ ContentRouter → SmartCrusher (JSON)
                   │                → CodeCompressor (AST)
                   │                → Kompress-base (文本, HF模型)
                   └─ CCR (可逆压缩)
```

**核心特性**：

| 特性 | 说明 |
|------|------|
| **多部署模式** | Library (Python/TS)、Proxy、Agent Wrap、MCP Server |
| **可逆压缩 (CCR)** | 原始内容本地缓存，LLM 可通过 `headroom_retrieve` 按需取回 |
| **6 种压缩算法** | SmartCrusher (JSON)、CodeCompressor (AST 感知)、Kompress-base (HF 模型)、Image、CacheAligner、IntelligentContext |
| **跨 Agent 记忆** | 共享存储，Claude/Codex/Gemini 共享，自动去重 |
| **headroom learn** | 从失败会话中挖掘经验，写入 CLAUDE.md / AGENTS.md |
| **多 Agent 兼容** | Claude Code、Codex、Cursor、Aider、Copilot CLI、OpenClaw |

**实测效果**：

| 工作负载 | 压缩前 | 压缩后 | 节省 |
|----------|--------|--------|------|
| Code search (100 results) | 17,765 tokens | 1,408 tokens | **92%** |
| SRE 事故调试 | 65,694 tokens | 5,118 tokens | **92%** |
| GitHub Issue 分类 | 54,174 tokens | 14,761 tokens | **73%** |
| 代码库探索 | 78,502 tokens | 41,254 tokens | **47%** |

**基准测试质量保持**：

| 基准 | 类别 | Baseline | Headroom | 差异 |
|------|------|----------|----------|------|
| GSM8K | 数学 | 0.870 | 0.870 | **±0.000** |
| TruthfulQA | 事实 | 0.530 | 0.560 | **+0.030** |
| SQuAD v2 | 问答 | — | 97% | 19% 压缩 |
| BFCL | 工具 | — | 97% | 32% 压缩 |

> **关键结论**：Headroom 在 GSM8K（数学）上零质量损失，TruthfulQA 甚至略有提升。证明了"压缩不降低质量"是可行的。

**与 ccmvp 体系的对比**：

| 维度 | Headroom | ccmvp (tokenforge+lean-ctx) |
|------|----------|---------------------------|
| 架构位置 | 代理层（外部）| 钩子层（内部）|
| 可逆性 | ✅ CCR 可逆压缩 | ❌ 不可逆 |
| AST 感知 | ✅ 多语言 AST | ❌ 仅正则匹配 |
| ML 模型 | ✅ Kompress-base (HF) | ❌ 无 |
| 跨 Agent 记忆 | ✅ | ❌ |
| 部署复杂度 | 需安装 pip/npm 包 | 纯 Node.js，零依赖 |
| 学习机制 | ✅ headroom learn | ❌ 无 |

### 3.2 LLMLingua ⭐ 5,000+

**仓库**：[microsoft/LLMLingua](https://github.com/microsoft/LLMLingua)

**定位**：微软研究院出品的 Prompt 压缩框架，EMNLP'23 + ACL'24 论文

**核心思路**：
- **LLMLingua-1**：粗到细的 prompt 压缩，通过 budget controller 控制压缩率
- **LLMLingua-2**：基于 GPT-4 数据蒸馏训练的小模型（BERT 级编码器），做 token 级别分类（保留/丢弃）
- 可将 prompt 压缩至原来的 1/20，同时保留 ICL（上下文学习）和推理能力

**优势**：
- 学术验证充分，有标准 benchmark
- 基于 ML 模型，压缩质量高
- 支持长文本（LongLLMLingua 变体）

**局限**：
- 需要额外的 ML 推理开销
- 对代码场景支持不如 Headroom
- 不是"零质量损失"——在极端压缩率下有信息丢失

### 3.3 RTK (Rust Token Killer) ⭐ 1,500+

**仓库**：[rtk-ai/rtk](https://github.com/rtk-ai/rtk)

**定位**：CLI 代理，压缩命令输出，60-90% Token 节省

**特点**：
- 单 Rust 二进制，零依赖
- 专精 CLI 命令输出压缩
- 被 Headroom 作为一级组件集成（`HEADROOM_CONTEXT_TOOL=lean-ctx` 或内置 RTK）

**典型场景**：
- `git show --short` → 摘要化
- `ls` → 紧凑格式
- `npm install` → 关键信息提取

### 3.4 lean-ctx ⭐ 360+

**仓库**：[yvgude/lean-ctx](https://github.com/yvgude/lean-ctx)

**定位**：上下文智能层，运行在 Agent 和 LLM 之间

**特点**：
- 71 MCP 工具、10 种读取模式、95+ 压缩模式
- Shell Hook + Claude Code Hook 双通道
- CCP（Context Continuity Protocol）跨会话持久化
- 单二进制部署，npm 安装

**与 ccmvp 的关系**：ccmvp 直接将 lean-ctx 作为其"基础设施技能"层的核心组件，通过 `settings.json` 的 MCP 配置集成。

### 3.5 Claw Compactor (open-compress)

**仓库**：[open-compress/claw-compactor](https://github.com/open-compress/claw-compactor)

**定位**：14 阶段 Fusion Pipeline 的 LLM Token 压缩引擎

**特点**：
- 14 阶段专业化压缩管线（可逆压缩、AST 感知代码分析、智能内容路由）
- 零 LLM 推理成本（不调用模型做压缩）
- MIT 开源

## 5. 极致节能体系架构建议

### 5.1 理想架构：「ccmvp 内核 + Headroom 增强」

综合 ccmvp（1234567KF）的 Pipeline 化节能理念和 Headroom 的工程化压缩能力，建议形成如下分层体系：

```
┌─────────────────────────────────────────────────────────┐
│                  极致节能五层架构                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  第 5 层：智能调度层                                       │
│  ┌───────────────────────────────────────────────────┐  │
│  │ kf-model-router + CCP Skip + skill-loader           │  │
│  │ (借鉴 ccmvp：任务复杂度感知 + 阶段跳过 + 按需加载)      │  │
│  └───────────────────────────────────────────────────┘  │
│                         ↓                                │
│  第 4 层：缓存复用层                                       │
│  ┌───────────────────────────────────────────────────┐  │
│  │ L1 共享前缀 + L2 长文预热 + L3 按需加载                │  │
│  │ + CacheAligner (借鉴 Headroom 的前缀稳定技术)          │  │
```

### 5.2 核心原则（来自两个海报的六大原则）

从「三角色海报」中提炼的 **Token 节能六大原则**：

1. **压缩即默认**：所有 CLI 输出默认经过 tokenforge/lean-ctx 压缩，用户无需手动操作
2. **缓存即杠杆**：利用 API Provider 原生 KV Cache，120x 的成本差价是最大的节约来源
3. **按需即智能**：非当前阶段技能仅保留元数据 stub，按需加载完整内容
4. **跳过即节约**：小变更自动跳过冗余阶段，每跳过一段省 10K-15K tokens
5. **压缩通信**：Agent 间通信使用压缩格式（λ-lang），而非冗长 JSON
6. **闭环优化**：监控 → 分析 → 优化 → 验证，持续迭代节能策略

### 5.3 实施路线图

### 5.4 通用性说明

以上所有技术方案（tokenforge、lean-ctx、KV Cache 策略、Headroom、RTK）均可在 **Claude Code** 和 **Qoder** 双平台通用：

- **tokenforge** → 纯 Node.js，通过 Hook/MCP 接入任意 Agent 平台
- **lean-ctx** → MCP Server 标准协议，跨平台兼容
- **Headroom** → Proxy 模式零代码侵入，Library 模式任意集成
- **KV Cache** → 依赖 API Provider，与 Agent 平台无关
- **RTK** → 单二进制，无平台依赖

## 6. 附录：技术实现细节

### 6.1 tokenforge.cjs 核心压缩算法

```javascript
// 输出压缩 (compressOutput)
1. ANSI 转义序列剥离  ────→ 移除终端颜色/格式代码
2. 连续空行折叠        ────→ 多个空行→保留1个
3. 重复行去重          ────→ 测试框架重复输出→计数显示
4. 栈跟踪折叠          ────→ 连续 at xxx 行→3行摘要
5. 长行截断            ────→ >300字符的行→297+...
6. 总行数限制          ────→ light:300/medium:120/aggressive:60

// 代码压缩 (compressCode)
1. 块注释移除          ────→ /* ... */ → 跳过
2. 导入/导出保留       ────→ import/export/require → 始终保留
3. 函数签名保留        ────→ function/class/def/fn → 保留
4. 函数体折叠          ────→ { 到 } 内代码→"N body lines folded"
5. 总行数限制          ────→ light:80/medium:40/aggressive:15

// JSON压缩 (compressJSON)
1. 深度限制            ────→ light:8/medium:5/aggressive:3
2. 数组采样            ────→ 长数组→取首尾 + 计数
3. 大对象摘要          ────→ >30键的对象→键列表摘要
```

### 6.2 DeepSeek KV Cache 缓存命中判定

```python
# 从 API 响应中读取缓存指标
usage.prompt_cache_hit_tokens    # 命中缓存的 prompt token 数
usage.prompt_cache_miss_tokens   # 未命中的 prompt token 数

cache_hit_rate = hit_tokens / (hit_tokens + miss_tokens)

# 命中率分级
> 70%  → 优秀，维持现状
30-70% → 一般，检查前缀一致性
< 30%  → 差，触发优化提示
```

### 6.3 Headroom 可逆压缩 (CCR) 机制

```
原始上下文 ──→ Headroom 压缩 ──→ 发送给 LLM
                  │
                  └──→ 原始内容存入本地缓存 (CCR Store)
                  │
LLM 需要更多信息时:
  headroom_retrieve(key) ──→ 从缓存取回原始内容 ──→ 发送给 LLM
```

这是 Headroom 与 tokenforge/lean-ctx 最本质的区别——**可逆压缩意味着零信息丢失**，LLM 始终可以按需获取完整内容。
